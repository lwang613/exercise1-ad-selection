## What I understand now
- The whole field reduces to one detective problem: *is this locus more extreme than a drift-and-demography-only world would produce?* Every statistic is a way of asking that.
- A reproducible project = immutable raw data + scripts that regenerate everything + version control. Environments (conda) and Git are the two halves of that discipline.
- Why my tools behave the way they do: compiled binaries vs interpreted code, CPU architectures (ARM64 vs x86_64), and how `PATH` lets the shell find commands.

## Decisions & rationale
- **Miniforge over Anaconda/Miniconda** — avoids the commercial `defaults` channel and its licensing terms on an institutional machine.
- **Build the env as `osx-64` under Rosetta** — a conda env is single-architecture, and a key tool (`plink2`) has no Apple-Silicon build, so Intel-under-Rosetta gives the complete toolset with negligible overhead.
- **Keep the repo private**, with `data/` excluded from Git — track the recipe, never the multi-GB VCFs.
- **Separate the lab notebook (this repo) from the Learning Notes vault** — a project diary vs a cross-project knowledge base.

## Problems & fixes

- **`curl` couldn't save the Miniforge installer:** I was sitting in the read-only root directory `/`. → Fix: `cd ~` first, then download.
